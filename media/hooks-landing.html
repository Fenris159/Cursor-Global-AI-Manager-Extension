Hooks

Hooks let you run custom scripts when the agent does certain things—for example when a session starts, before or after the agent edits a file, runs a shell command, or uses an MCP tool. You can use them to log activity, enforce checks, or change how the agent behaves. This extension manages your user-level ~/.cursor/hooks.json (which events run which scripts) and the script files in ~/.cursor/hooks/.

How to manage hooks with this extension

Expand the Hooks section in the sidebar. You'll see:

  • hooks.json — The configuration file that defines which scripts run at each event. Select it to view or edit. Use the trash icon to clear all hooks (empty config).
  • hooks (folder) — Expand to see your hook scripts. Each script has a checkbox to enable or disable it for its event. Use + Create new to add a stub script, or Import (folder icon in the section header) to add an existing script from disk.

Select a script to preview and edit it. Use Edit to open it in the editor. Use Sync to Workspace or Export to copy hooks into a project or elsewhere. Once you enable a script (check the box), it runs automatically whenever the agent triggers that event—no need to run a command or restart Cursor.

Placeholders: first run and respawn

First run after install. The very first time you expand the Hooks section after installing this extension, the extension will:

  • Create hooks.json with empty hooks if it doesn't exist (no scripts registered).
  • Create the 20 default hook script files in ~/.cursor/hooks/ (e.g. after-file-edit.sh, session-start.sh) as stub scripts if they are missing. These are added as files only; none are enabled in hooks.json, so they won't run until you check them.

This happens only once per install. On later sessions, expanding Hooks just loads existing data and does not create or overwrite files.

Respawn placeholders. If you delete some or all of the default hook scripts and want them back as empty stubs, use Spawn placeholders in the hooks folder (under + Create new). It creates only the default hook files that are currently missing; existing files are left unchanged. New files are created unchecked so they do not get added to hooks.json.

Script naming and event mapping

Script filenames map to hook events. Use kebab-case matching the event name: after-file-edit.sh → afterFileEdit, before-shell-execution.js → beforeShellExecution. Supported extensions: .sh, .js, .ts. When you enable a script via the checkbox, the extension registers it in hooks.json for the matching event.

Multiple scripts per event

You can enable more than one script for the same hook event (e.g. two scripts for afterFileEdit). Cursor runs all of them for that event; they all execute.

Duplicate extensions (advanced)

If you use multiple Cursor extensions or install the same extension in both user and workspace, Cursor follows VS Code's model:

SituationWhat happens
Same extension ID (user + workspace)Workspace version wins
Different extension IDsBoth activate
Same command IDLast registered wins (can cause override issues)
Same event hook (multiple scripts)All scripts for that event run

To inspect what's running: open Help → Toggle Developer Tools, then check Running Extensions and activation events. You can also add logging in your hook scripts to confirm when they run.

Script contract

Each hook script receives input from Cursor and can optionally return output. Follow this contract so Cursor runs your script correctly:

  • Input — JSON passed via stdin (event-specific payload).
  • Output — Optional JSON via stdout; only some hooks (e.g. before*) use it to block or modify the action.
  • Exit 0 — Success; the agent continues.
  • Exit 2 — Block the action. Only meaningful for before* hooks (e.g. beforeShellExecution); Cursor will cancel the pending action. Other hooks should use exit 0.
Windows. .sh scripts require Bash (Git Bash, WSL, etc.). For cross-platform hooks, use .js or .ts — the extension registers them with node ./hooks/your-script.js so they run on any OS.
